1. Overview



1.1 Problem Statement

Managing scholarship and opportunity pipelines is effectively a part-time job. The current state involves scattered browser bookmarks, missed deadlines, and duplicated effort when sharing opportunities with junior peers. There is no single system that handles discovery, triage, application tracking, and junior distribution in one place.



1.2 Solution

A Manifest V3 Chrome Extension paired with Notion as a backend database. The extension handles rapid, zero-friction data capture directly on scholarship pages. Notion serves as both the data store and the UI for deeper organizational workflows, including a curated junior distribution portal.



1.3 Goals

Reduce time-to-capture of a new opportunity to under 10 seconds

Eliminate missed deadlines by surfacing 7-day countdowns in the extension popup

Enable one-click forwarding of vetted opportunities to a juniors-facing Notion portal

Track full application lifecycle from discovery to confirmation received

Provide a CRM-style layer for networking contacts attached to each opportunity



1.4 Non-Goals

Automatically writing or submitting applications

Scraping opportunity listings from external job boards

Building a standalone web dashboard (Notion is the UI)

Mobile support in V1



2. User Personas



2.1 Primary User: The Builder

A final-year software engineering student actively hunting scholarships, fellowships, grants, and hackathons. Technically proficient, works in sprint-like bursts, values speed and resource efficiency. Wants a system that does the busywork while they stay in flow.



Browses 10–20+ opportunity pages per week

Manages multiple active applications simultaneously

Occasionally applies at the last minute and needs deadline panic-alerts

Mentors juniors and wants to forward relevant opportunities without extra friction



2.2 Secondary User: The Junior

A first- or second-year student with limited time and discovery reach. They don’t hunt; they rely on the primary user to surface pre-vetted opportunities. They consume from the shared Notion portal and need clean, filtered results with no noise from the primary user’s private workflow.

5. Notion Architecture



5.1 Database Schema

The Notion database maps 1:1 to the JSON schema above. Each group (Opportunity Info, Triage, Execution, CRM, Junior) maps to a Notion property group. Property types are selected to match the field types in the schema.



5.2 Workflow: Personal Command Center

View: Kanban board grouped by Status property

Use: Cards move left-to-right as the application progresses

Filter: A saved “Immediate Action” view shows only cards where Next Action Date ≤ Today

Filter: A saved “Blocked” view shows cards where Current Blocker is not empty

Template: A database template auto-populates a Required Assets checklist for each new card



5.3 Workflow: Junior Distribution Portal

Type: A Linked Database View on a separate shared/public Notion page

Permanent filter: [Share with Juniors] is Checked AND [Final Deadline] is after Today

Visible columns: Name, Organization, Value, Type, Final Deadline, Eligibility Level, Geographic Requirements

Hidden columns: All personal tracking fields (Status, Blocker, Working Doc, CRM fields)

Juniors see curated gold. They never see your personal workflow or private notes.



6. Technical Architecture



6.1 Extension Stack

Extension Standard

Manifest V3 (required for modern Chrome)

Popup UI

TypeScript + React + Tailwind CSS

State Management

chrome.storage.local (cache active Next Action Dates)

Content Script

Vanilla TS injected into active tab for DOM scraping

Background Worker

Service Worker — handles auth, API calls, and CORS avoidance

API Client

Fetch API calling Notion REST v1

Auth

Notion Integration Token stored in extension environment, never DOM-exposed



6.2 Content Script Behavior

The content script is injected into the active tab when the popup is opened. It extracts:

document.title → opportunity name (editable before save)

window.location.href → URL

Regex scan of visible text for common date formats: MM/DD/YYYY, DD Month YYYY, Month DD YYYY, “Deadline: X”, “Close: X”

Scraped data is sent via chrome.runtime.sendMessage to the popup. The popup pre-fills fields; the user edits if needed and saves.



6.3 Background Service Worker

The service worker is responsible for all outbound API calls. This prevents CORS errors that occur when hitting the Notion API from a page’s execution context. It:

Receives the structured payload from the popup

Attaches the Notion Integration Token from chrome.storage.local

POSTs to https://api.notion.com/v1/pages with the target database ID

Returns success/failure status to the popup for UI feedback



6.4 Notion API Integration

Endpoint

POST https://api.notion.com/v1/pages

Auth Header

Authorization: Bearer <NOTION_INTEGRATION_TOKEN>

Notion-Version

2022-06-28

Setup

Create Internal Integration in Notion → share target DB with integration → copy Integration Token

Error Handling

Popup displays inline error banner on 4xx/5xx. Retries not automatic in V1.



6.5 Security Constraints

Notion Integration Token is stored in chrome.storage.local, never hardcoded in source

Token is never injected into or accessible from the DOM of scraped pages

No external analytics or telemetry in V1

All scraped data is user-visible before submission. No silent background scraping.



7. UX & Interaction Requirements



7.1 Popup Layout

The popup is a single scrollable panel, approximately 380px wide. It is divided into collapsible sections:

Header: Page title (auto-scraped), editable, with Save and Discard actions

Triage: Status, Priority, Final Deadline, Next Action Date, Effort Level

Execution: Current Blocker, Working Doc Link, Review Stage, Required Assets

CRM: POC Name, POC Email, Outreach Status, Follow-up Date

Distribution: Junior Toggle, Eligibility Tags, Geo Tags

Footer: Triage Dashboard — 7-day countdown cards for upcoming Next Action Dates



7.2 Performance Requirements

Popup must render in < 200ms (no network call on open; use cached chrome.storage.local data)

Notion API POST must complete in < 3 seconds. Show loading spinner. Show error on timeout.

DOM scraping must complete in < 500ms. Degrade gracefully if no date found.



7.3 Accessibility

All interactive elements are keyboard-navigable

Color is not the sole indicator of status (labels accompany color coding)

Minimum touch target size: 32px for all buttons



8. Out of Scope (V1)



Automatic opportunity discovery or listing scraping from external boards

Multi-browser support (Firefox, Safari) — Chrome only in V1

Mobile companion app

Automated follow-up email sending

Essay writing assistance or AI prompting within the extension

Team collaboration beyond the junior portal (no multi-user editing of the same pipeline)

Offline mode (requires network for Notion API sync)
