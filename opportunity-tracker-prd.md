Overview</strong></h1><h2><strong>1.1 Problem Statement</strong></h2><p>Managing scholarship and opportunity pipelines is effectively a part-time job. The current state involves scattered browser bookmarks, missed deadlines, and duplicated effort when sharing opportunities with junior peers. There is no single system that handles discovery, triage, application tracking, and junior distribution in one place.</p><h2><strong>1.2 Solution</strong></h2><p>A Manifest V3 Chrome Extension paired with Notion as a backend database. The extension handles rapid, zero-friction data capture directly on scholarship pages. Notion serves as both the data store and the UI for deeper organizational workflows, including a curated junior distribution portal.</p><h2><strong>1.3 Goals</strong></h2><ul><li>Reduce time-to-capture of a new opportunity to under 10 seconds</li><li>Eliminate missed deadlines by surfacing 7-day countdowns in the extension popup</li><li>Enable one-click forwarding of vetted opportunities to a juniors-facing Notion portal</li><li>Track full application lifecycle from discovery to confirmation received</li><li>Provide a CRM-style layer for networking contacts attached to each opportunity</li></ul><h2><strong>1.4 Non-Goals</strong></h2><ul><li>Automatically writing or submitting applications</li><li>Scraping opportunity listings from external job boards</li><li>Building a standalone web dashboard (Notion is the UI)</li><li>Mobile support in V1</li></ul><h1><strong>2. User Personas</strong></h1><h2><strong>2.1 Primary User: The Builder</strong></h2><p>A final-year software engineering student actively hunting scholarships, fellowships, grants, and hackathons. Technically proficient, works in sprint-like bursts, values speed and resource efficiency. Wants a system that does the busywork while they stay in flow.</p><ul><li>Browses 10–20+ opportunity pages per week</li><li>Manages multiple active applications simultaneously</li><li>Occasionally applies at the last minute and needs deadline panic-alerts</li><li>Mentors juniors and wants to forward relevant opportunities without extra friction</li></ul><h2><strong>2.2 Secondary User: The Junior</strong></h2><p>A first- or second-year student with limited time and discovery reach. They don’t hunt; they rely on the primary user to surface pre-vetted opportunities. They consume from the shared Notion portal and need clean, filtered results with no noise from the primary user’s private workflow.</p><h1><strong>3. Notion Architecture</strong></h1><h2><strong>5.1 Database Schema</strong></h2><p>The Notion database maps 1:1 to the JSON schema above. Each group (Opportunity Info, Triage, Execution, CRM, Junior) maps to a Notion property group. Property types are selected to match the field types in the schema.</p><h2><strong>5.2 Workflow: Personal Command Center</strong></h2><ul><li>View: Kanban board grouped by Status property</li><li>Use: Cards move left-to-right as the application progresses</li><li>Filter: A saved “Immediate Action” view shows only cards where Next Action Date ≤ Today</li><li>Filter: A saved “Blocked” view shows cards where Current Blocker is not empty</li><li>Template: A database template auto-populates a Required Assets checklist for each new card</li></ul><h2><strong>5.3 Workflow: Junior Distribution Portal</strong></h2><ul><li>Type: A Linked Database View on a separate shared/public Notion page</li><li>Permanent filter: [Share with Juniors] is Checked AND [Final Deadline] is after Today</li><li>Visible columns: Name, Organization, Value, Type, Final Deadline, Eligibility Level, Geographic Requirements</li><li>Hidden columns: All personal tracking fields (Status, Blocker, Working Doc, CRM fields)</li><li>Juniors see curated gold. They never see your personal workflow or private notes.</li></ul><h1><strong>6. Technical Architecture</strong></h1><h2><strong>6.1 Extension Stack</strong></h2><table><tr><td><p><strong>Extension Standard</strong></p></td><td><p>Manifest V3 (required for modern Chrome)</p></td></tr><tr><td><p><strong>Popup UI</strong></p></td><td><p>TypeScript + React + Tailwind CSS</p></td></tr><tr><td><p><strong>State Management</strong></p></td><td><p>chrome.storage.local (cache active Next Action Dates)</p></td></tr><tr><td><p><strong>Content Script</strong></p></td><td><p>Vanilla TS injected into active tab for DOM scraping</p></td></tr><tr><td><p><strong>Background Worker</strong></p></td><td><p>Service Worker — handles auth, API calls, and CORS avoidance</p></td></tr><tr><td><p><strong>API Client</strong></p></td><td><p>Fetch API calling Notion REST v1</p></td></tr><tr><td><p><strong>Auth</strong></p></td><td><p>Notion Integration Token stored in extension environment, never DOM-exposed</p></td></tr></table><h2><strong>6.2 Content Script Behavior</strong></h2><p>The content script is injected into the active tab when the popup is opened. It extracts:</p><ul><li>document.title → opportunity name (editable before save)</li><li>window.location.href → URL</li><li>Regex scan of visible text for common date formats: MM/DD/YYYY, DD Month YYYY, Month DD YYYY, “Deadline: X”, “Close: X”</li></ul><p>Scraped data is sent via chrome.runtime.sendMessage to the popup. The popup pre-fills fields; the user edits if needed and saves.</p><h2><strong>6.3 Background Service Worker</strong></h2><p>The service worker is responsible for all outbound API calls. This prevents CORS errors that occur when hitting the Notion API from a page’s execution context. It:</p><ul><li>Receives the structured payload from the popup</li><li>Attaches the Notion Integration Token from chrome.storage.local</li><li>POSTs to https://api.notion.com/v1/pages with the target database ID</li><li>Returns success/failure status to the popup for UI feedback</li></ul><h2><strong>6.4 Notion API Integration</strong></h2><table><tr><td><p><strong>Endpoint</strong></p></td><td><p>POST https://api.notion.com/v1/pages</p></td></tr><tr><td><p><strong>Auth Header</strong></p></td><td><p>Authorization: Bearer &lt;NOTION_INTEGRATION_TOKEN&gt;</p></td></tr><tr><td><p><strong>Notion-Version</strong></p></td><td><p>2022-06-28</p></td></tr><tr><td><p><strong>Setup</strong></p></td><td><p>Create Internal Integration in Notion → share target DB with integration → copy Integration Token</p></td></tr><tr><td><p><strong>Error Handling</strong></p></td><td><p>Popup displays inline error banner on 4xx/5xx. Retries not automatic in V1.</p></td></tr></table><h2><strong>6.5 Security Constraints</strong></h2><ul><li>Notion Integration Token is stored in chrome.storage.local, never hardcoded in source</li><li>Token is never injected into or accessible from the DOM of scraped pages</li><li>No external analytics or telemetry in V1</li><li>All scraped data is user-visible before submission. No silent background scraping.</li></ul><h1><strong>7. UX &amp; Interaction Requirements</strong></h1><h2><strong>7.1 Popup Layout</strong></h2><p>The popup is a single scrollable panel, approximately 380px wide. It is divided into collapsible sections:</p><ul><li>Header: Page title (auto-scraped), editable, with Save and Discard actions</li><li>Triage: Status, Priority, Final Deadline, Next Action Date, Effort Level</li><li>Execution: Current Blocker, Working Doc Link, Review Stage, Required Assets</li><li>CRM: POC Name, POC Email, Outreach Status, Follow-up Date</li><li>Distribution: Junior Toggle, Eligibility Tags, Geo Tags</li><li>Footer: Triage Dashboard — 7-day countdown cards for upcoming Next Action Dates</li></ul><h2><strong>7.2 Performance Requirements</strong></h2><ul><li>Popup must render in &lt; 200ms (no network call on open; use cached chrome.storage.local data)</li><li>Notion API POST must complete in &lt; 3 seconds. Show loading spinner. Show error on timeout.</li><li>DOM scraping must complete in &lt; 500ms. Degrade gracefully if no date found.</li></ul><h2><strong>7.3 Accessibility</strong></h2><ul><li>All interactive elements are keyboard-navigable</li><li>Color is not the sole indicator of status (labels accompany color coding)</li><li>Minimum touch target size: 32px for all buttons</li></ul><h1><strong>8.
